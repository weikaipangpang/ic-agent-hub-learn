# 前端设计流程指南
本规范定义完整的芯片前端设计流水线，统一管控各阶段流转与衔接。

## 适用边界
- 不执行各阶段专属的深度分析与代码生成工作，仅负责流程编排与阶段顺序管控。
- 流程编排层不做架构决策、不产出设计代码。
- 问题出现时果断回滚；前置条件满足时稳步推进。

本文档用途：
- 了解完整流程阶段顺序及各阶段输出产物
- 从任意节点启动并跑完完整设计流水线
- 在中途接入项目时，快速定位当前所处阶段
- 当下游阶段发现上游问题时，执行流程回滚处理

## 流水线阶段总览
| 阶段 | 功能模块 | 输入 | 输出 | 下一阶段 |
|------|----------|------|------|----------|
| 1 | `前端规格拆解` | 需求规格、协议文档、寄存器文档 | 接口、寄存器、参数、时钟/复位汇总文档 | 阶段2 |
| 2 | `前端规格评审` | 阶段1输出文件 | 就绪/风险/阻塞 结论判定 | 就绪则进入阶段3 |
| 3 | `前端微架构分析` | 阶段1规格拆解结果+阶段2评审结论+性能/功耗约束 | 微架构概要、架构设计约束 | 阶段4 |
| 4 | `前端RTL架构规划` | 阶段2评审结论+阶段3微架构输出 | RTL整体框架、模块层级、编码实现顺序 | 阶段5、6并行 |
| 5 | `前端RTL代码生成` | 阶段4架构方案+阶段1需求规格 | 存放于`rtl/`目录的SV设计源码 | 阶段7 |
| 6 | `前端测试用例生成` | 阶段1需求规格+阶段4架构端口定义 | 存放于`tb/`目录的SV测试平台代码 | 阶段7 |
| 7 | `前端集成与环境初始化` | 阶段5 RTL代码+阶段6测试平台代码 | 文件列表、集成评审报告、编译脚本规划 | 阶段8 |
| 8 | `前端EDA环境配置` | 阶段7集成方案+本地EDA工具环境 | 仿真编译脚本、工具清单、波形配置文件 | 阶段9 |
| 9 | `编译与仿真运行` | 仿真编译脚本+文件列表 | 编译/仿真结果、运行日志记录 | 流程结束 或 阶段10 |
| 10 | `问题分类与初步定位` | 编译日志 / 仿真日志 | 故障分类、修复建议 | 阶段9 或 阶段11 |
| 11 | `波形调试分析` | 阶段10定位结论+波形转储文件 | 波形分析结论、问题根因 | 阶段9 |

## 项目配置规范
所有项目根目录必须存在`input_config.json`配置文件。
若仅提供需求规格文档，阶段1（前端规格拆解）将基于模板自动生成带默认参数的配置文件。
所有路径统一使用**项目相对路径**，禁止绝对路径。

配置模板参考：[input_config.template.json](assets/input_config.template.json)

配置字段说明：
| 字段 | 必填 | 说明 |
|------|------|------|
| `version` | 是 | 配置文件版本（当前固定为`"1.1"`） |
| `design_name` | 是 | 设计短名称，用于各类产物文件命名 |
| `spec.files` | 是 | 需求规格文档相对路径列表 |
| `spec.format` | 是 | 文档格式：`markdown`/`pdf`/`docx` |
| `rtl.dir` | 是 | RTL源码存放目录（相对路径） |
| `rtl.top_module` | 是 | 顶层设计模块名 |
| `rtl.include_dirs` | 否 | 编译额外头文件包含目录 |
| `rtl.defines` | 否 | 编译预处理宏定义 |
| `rtl.filelist` | 否 | 自定义文件列表路径，存在则阶段7不再自动生成 |
| `tb.dir` | 是 | 测试平台代码存放目录（相对路径） |
| `tb.top_module` | 是 | 测试平台顶层模块名 |
| `sim.dir` | 是 | 仿真工作目录（相对路径） |
| `sim.simulator` | 是 | 仿真器：`auto`/`vcs`/`xcelium`/`questa` |
| `sim.waveform` | 否 | 波形查看工具：`auto`/`verdi`/`dve`/`simvision` |
| `sim.dump_format` | 否 | 波形存储格式：`auto`/`fsdb`/`vpd`/`shm` |
| `sim.timescale` | 否 | 仿真时间精度，默认`1ns/1ps` |
| `target.platform` | 否 | 目标平台：`asic`/`fpga`，影响性能约束设计 |
| `target.clock_freq_mhz` | 否 | 规格定义目标时钟频率（兆赫兹） |
| `design_options` | 否 | 设计自定义键值参数 |
| `docs.dir` | 是 | 各阶段产物文档输出目录（相对路径） |

## 阶段产物读写规范
| 阶段 | 读取文件 | 输出文件 |
|------|----------|----------|
| 1 | 规格文档 + 配置文件（缺失则自动生成） | 自动补全配置文件 + 阶段1拆解报告 |
| 2 | 阶段1拆解报告 | 阶段2规格评审报告 |
| 3 | 阶段1拆解报告 + 阶段2评审报告 | 阶段3微架构分析报告 |
| 4 | 阶段2评审报告 + 阶段3分析报告 | 阶段4 RTL架构规划报告 |
| 5 | 阶段4架构方案、可选微架构文档、阶段1规格 | 阶段5生成记录 + RTL设计代码 |
| 6 | 阶段1规格文档 + 阶段4架构端口定义 | 阶段6生成记录 + 测试平台代码 |
| 7 | RTL代码、测试平台代码、架构规划文档 | 集成方案报告 + 仿真文件列表 |
| 8 | 阶段7集成方案 + EDA环境信息 | EDA环境配置报告、工具检测文件、仿真编译脚本 |
| 9 | 仿真编译脚本 + 文件列表 | 仿真运行记录 + 各类日志文件 |
| 10 | 仿真运行记录 + 日志文件 | 故障分类与定位报告 |
| 11 | 问题定位报告 + 波形文件 | 波形调试分析报告 |

通用规则：
1. 每个阶段启动前，必须校验前置输入产物是否完整存在。
2. 阶段完成前，必须输出本阶段标准产物文件。
3. 产物文件名统一固定命名，禁止自定义修改。
4. 若文档目录不存在，阶段自动创建目录。

## 阶段流转与交付逻辑
```
需求规格 --> [1.规格拆解] --> [2.规格评审] --> [3.微架构分析] --> [4.RTL架构规划]
             ↑             |  ↑                               |
             |     评审阻塞 |  | 问题修复               +-----+-----+
             |             ↓  |                       |           |
             |          [1.规格拆解]              [5.RTL编码] [6.测试平台开发]
             |                                         |           |
             |                                         +-----+-----+
             |                                               |
             |                                        [7.代码集成]
             |                                               |
             |                                        [8.EDA环境部署]
             |                                               |
             |          +--------- 问题修复 --------+  [9.编译仿真]
             |          ↓                          |    |  ↑  |
             +---- [10.问题初步定位] ----+          |    |  |  | 仿真失败
                        |                |          |    |  +--+
                需要波形调试             +----------+    |
                        ↓                               |
                [11.波形深度调试] ----> 修复 ----> [9.编译仿真]
                        ↑                               |
                        |        仿真通过后自检          |
                        +---------- 健康检查 -----------+
                                    经由阶段10完成
```
所有仿真运行结果（通过/失败）都必须经过阶段10、阶段11闭环，确保全流程规范落地执行。

### 阶段2 核心门禁
阶段2规格评审为流程关键卡点：
- `ready（就绪）`：正常进入阶段3
- `risky（存在风险）`：携带风险标记进入阶段3
- `blocked（流程阻塞）`：退回阶段1重新拆解规格

### 阶段5、6 并行机制
RTL编码与测试平台开发支持并行执行，二者无依赖关系：
- RTL编码：依赖架构方案+需求规格
- 测试平台：依赖规格行为定义+架构端口信息
- 两项工作全部完成后，才可进入阶段7代码集成

### 9-10-11 调试闭环与通过后自检
**仿真失败场景**：
阶段9自动进入阶段10日志级问题定位；日志无法定位时，转入阶段11波形调试。问题修复后回到阶段9重跑，最大允许3轮迭代，无法解决则升级上报。

**仿真通过场景**：
仍需执行阶段10后置健康检查（排查编译/仿真告警、隐性问题）+阶段11抽样波形核验，确认关键信号时序与逻辑正常，保障交付质量。

## 安全防护机制
### 阶段5/6 代码生成前置校验
| 检查项 | 执行规则 |
|--------|----------|
| RTL/测试目录已存在文件 | 覆盖前自动备份至时间戳命名的备份目录 |
| 备份异常失败 | 立即终止流程，禁止强制覆盖文件 |

### 阶段8 EDA环境配置前置校验
| 检查项 | 执行规则 |
|--------|----------|
| 已有编译脚本、仿真器一致 | 跳过脚本重新生成 |
| 已有编译脚本、仿真器变更 | 先备份仿真目录，再清理重建配置 |
| 备份异常失败 | 终止操作 |

### 阶段9 每次编译前置校验
| 检查项 | 执行规则 |
|--------|----------|
| 存在历史日志、覆盖率文件 | 编译前自动备份 |
| 首次初始化运行 | 跳过备份 |
| 备份异常失败 | 终止编译 |

### 备份范围规范
- 需备份：日志文件、覆盖率目录、覆盖率报告
- 无需备份：仿真产物、编译临时文件、超大波形文件（可重新生成）
- 目录格式：`备份目录/年-月-日_时-分-秒/`
- 强制规则：禁止手动删除历史备份目录

## 流程目标
- 清晰标识项目当前所处设计阶段
- 完整展示流水线顺序与跨阶段交付关系
- 显性化缺失的前置依赖产物
- 按需精准回滚，杜绝盲目推进流程

## 流程编排规则
1. 梳理项目现有产物与设计目标。
2. 根据阶段对照表，精准匹配当前所处阶段。
3. 校验上游阶段输出文件完整性与有效性。
4. 前置条件缺失/冲突时，退回对应上游阶段。
5. 依赖完备后，按顺序执行当前阶段工作。
6. 阶段5、6支持并行调度。
7. 单阶段完成后，通过门禁校验，流转至下一阶段。
8. 调试循环严格限制3轮上限，超阈值升级处理。

## 项目中途接入规则
| 现有资料 | 接入阶段 | 前置校验要求 |

| 微架构分析文档 | 阶段4 | 校验阶段3产物存在 |
| 架构规划方案 | 阶段5、6 | 校验阶段4产物存在 |
| RTL+测试平台完整代码 | 阶段7 | 设计与测试文件齐全 |
| 集成方案文档 | 阶段8 | 校验阶段7产物存在 |
| 编译脚本+文件列表 | 阶段9 | 仿真目录核心文件齐全 |

## 流程回滚规则
| 当前阶段 | 回滚触发条件 | 退回阶段 |

| 阶段9 编译仿真 | 脚本缺失/版本过期 | 阶段8 |
| 阶段9 编译仿真 | 源码/文件列表缺失 | 阶段7 |
| 阶段10 问题定位 | 设计与测试接口定义偏移 | 阶段7 |

| 阶段10 问题定位 | 原始需求规格问题 | 阶段1 或 阶段2 |

## 阶段完成门禁
本流程规范执行完成标准：
- 精准判定当前设计阶段
- 明确展示剩余全流程路径
- 完整列出缺失的前置依赖
- 清晰定义回滚与流程终止条件

## 可复用输出内容
- 当前阶段评估报告
- 剩余完整流水线执行计划
- 缺失依赖文件清单
- 流程回滚处理结论（如需）
- 下一阶段执行指令

## 总体决策准则
1. 严格按阶段顺序推进；中途接入需满足全部前置产物要求。
2. 阶段2为硬门禁，阻塞状态禁止进入后续设计环节。
3. 并行阶段必须全部完成，才可进入集成环节。
4. 仿真调试循环最多执行3轮，避免无限迭代。
5. 流程回滚优先退回最近的未完成上游阶段。
6. 编排层仅负责流程调度与交付管控，不替代各阶段专业设计工作。
